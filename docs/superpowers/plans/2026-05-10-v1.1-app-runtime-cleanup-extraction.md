# PaddockJS 1.1 App Runtime Cleanup and Extraction Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development when explicitly permitted, or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Continue the post-`v1.0.0` cleanup for `v1.1.0` by refactoring the browser app runtime around `src/app/F1SimulatorApp.js` into small, readable, feature-owned modules while preserving package behavior, public APIs, layout behavior, deterministic simulation behavior, and host integration.

**Architecture:** Keep `F1SimulatorApp` as the package-owned browser runtime facade and orchestration shell. Move camera framing, PixiJS rendering surfaces, DOM readout rendering, race-data/radio banners, steward messages, and runtime loop concerns into focused modules. The simulator core refactor is already complete; do not reopen simulation behavior unless an app extraction exposes a real integration bug.

**Tech Stack:** JavaScript ES modules, PixiJS, browser DOM APIs, Vitest, Playwright/browser smoke, PaddockJS package verification, portfolio consumer verification, Linear work logging.

---

## Release Context

`v1.0.0` is already released. This cleanup is for `v1.1.0`.

The previous cleanup extracted simulation logic from `src/simulation/raceSimulation.js`. The next bottleneck is now `src/app/F1SimulatorApp.js`, which remains the largest mixed-responsibility file in the package.

This plan should make the runtime:

- clean
- easy to read
- easy to modify
- easy to add new features
- modular
- divided into utility and feature-owned modules

## Current State

`src/app/F1SimulatorApp.js` is about 2,536 lines and currently owns all of these responsibilities:

- PixiJS app lifecycle
- asset loading
- car and safety-car sprite creation
- DRS trail rendering
- expert sensor ray rendering
- pit-lane status light rendering
- camera modes, zoom, safe-area measurement, and bounds fitting
- frame pacing and runtime visibility behavior
- DOM readout updates
- timing tower markup rendering
- telemetry and sector readout rendering
- car/driver overview rendering
- race-data project/radio lower-third behavior
- steward penalty/warning banner behavior
- restart and destroy orchestration
- host lifecycle callback dispatch

This is now the clearest "god file" in the codebase. Adding later v1.1 features here would increase coupling and make changes harder to verify.

## Non-Goals

- Do not change public package APIs.
- Do not change host mounting behavior.
- Do not change generated DOM contracts or data attributes unless a task explicitly says so.
- Do not redesign UI layout, timing-board width, camera proportions, or broadcast presentation.
- Do not change race rules, physics, DRS behavior, pit-stop behavior, snapshots, or deterministic simulation logic.
- Do not refactor `src/simulation/track/trackModel.js` in this plan. Track-model cleanup should be a separate plan after the app runtime is smaller.
- Do not treat generated portfolio build output as source cleanup.
- Do not add new v1.1 features in this cleanup pass.

## Success Criteria

By the end of this plan:

- `src/app/F1SimulatorApp.js` is an app facade/orchestrator, not a mixed rendering/control/readout implementation file.
- Camera logic is isolated and testable without reading the whole app class.
- PixiJS rendering surface logic is split by visual responsibility.
- DOM readout rendering is split by UI surface.
- Runtime frame loop and visibility behavior is separated from rendering details.
- Existing package tests pass.
- Package check passes.
- Portfolio consumer install/build passes.
- Browser smoke passes against the portfolio host page.
- Docs describe the new app-runtime ownership boundaries.
- Linear is updated with work done, verification, and follow-up.

## Constraints and Risks

- `F1SimulatorApp` has many implicit caches. Extract state carefully and keep cache ownership explicit.
- DOM readouts come from `querySimulatorDom(root)`; extracted modules should receive the relevant `readouts` object or specific nodes rather than querying global DOM.
- PixiJS display objects have lifecycle rules. Extracted renderer modules must avoid destroying shared textures.
- Browser smoke is required because the runtime can pass unit tests while still failing mounted UI behavior.
- Portfolio verification can update generated install/build artifacts. Do not confuse those with source refactor changes.
- The timing board width and broadcast proportions are package-owned. Do not retune them as part of cleanup.

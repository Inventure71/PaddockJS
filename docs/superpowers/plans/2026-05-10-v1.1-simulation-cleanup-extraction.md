# PaddockJS 1.1 Simulation Cleanup and Extraction Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prepare the already-released `v1.0.0` simulator codebase for `v1.1.0` by finishing the next cleanup/extraction layer around `src/simulation/raceSimulation.js` while preserving current behavior and leaving clear module boundaries for later feature work.

**Architecture:** Keep `F1RaceSimulation` as the stable public facade and deterministic step orchestrator. Move remaining feature-specific behavior out of the facade into focused modules for pit flow, rules review, contact resolution, runoff response, race snapshots, driver race context, and race-control construction. This plan is a cleanup foundation for later `v1.1.0` features; it should not add weather, reliability, fuel-load, custom-track, or new race-rule behavior.

**Tech Stack:** JavaScript ES modules, Vitest, PixiJS browser runtime, PaddockJS simulation/environment APIs, Linear work logging.

---

## Release Context

`v1.0.0` is already released. This plan is for `v1.1.0`.

The purpose is not to reopen the 1.0 modularization work. The purpose is to reduce the remaining responsibility load in `src/simulation/raceSimulation.js` so later feature work can land in explicit ownership modules instead of expanding the facade again.

## Current State

After the 1.0 extraction pass, `src/simulation/raceSimulation.js` is still about 1,381 lines and still owns several blocks of feature behavior:

- pit stop flow orchestration: scheduling, rearming, start, and advance flow
- rules review orchestration: calling every steward type directly
- physical contact resolution: collision correction and velocity response
- runoff response: off-track speed/heading correction
- race snapshot assembly: full, render, and observation snapshot assembly
- race order/context helpers: ordered cars, driver race context, aggression calculation
- race-control construction: constructor-owned initialization of race-control and safety-car state
- race-state recalculation: distance/progress/lap telemetry refresh, ranking fields, attack state, DRS update loop, and finish evaluation trigger
- race-finish transitions: car-finish events, final all-cars-finished transition, winner mode switch, safety-car deployment, and finish event emission
- utility helpers: lap normalization, race-distance wrapping, penalty-stat aggregation, legal pit-lane exemption checks, and collision-steward context helpers

The modules already created during the 1.0 cleanup should remain and be extended:

- `src/simulation/pit/`
- `src/simulation/race/`
- `src/simulation/rules/`
- `src/simulation/timing/`
- `src/simulation/vehicle/`
- `src/simulation/track/`

## Non-Goals

- Do not change public APIs.
- Do not change snapshot shape.
- Do not change deterministic behavior for identical seeds, track seeds, drivers, entries, rules, and controls.
- Do not add new race features in this cleanup plan.
- Do not remove compatibility re-exports such as `src/simulation/trackModel.js` unless a separate release plan says to do so.
- Do not treat generated portfolio build output as source cleanup.

## Target Ownership

### `src/simulation/raceSimulation.js`

Owns only:

- constructor wiring
- fixed-step ordering
- shared state containers
- public facade methods
- event buffer lifecycle
- calls into feature modules

Should not own after this plan:

- pit stop scheduling or flow internals
- direct steward review wiring
- collision/contact physics
- runoff correction math
- snapshot object assembly
- driver context/aggression logic
- race-control default object construction
- race-distance utility math
- penalty-stat aggregation
- detailed race-state recalculation
- finish transition details

### `src/simulation/pit/pitFlow.js`

Owns:

- detecting active pit stops
- scheduling at next pit entry
- rearming completed pit stops
- pit-entry eligibility checks
- deciding whether a pit stop should start
- starting a pit stop
- advancing a pit stop through entering, queue, service, and exit phases
- delegating service and routing details to `pitService.js` and `pitRouting.js`

### `src/simulation/rules/rulesReview.js`

Owns:

- collision steward review dispatch
- tire-requirement review dispatch
- track-limit review dispatch
- pit-lane speeding review dispatch
- translating steward results into `recordPenalty()` calls and race events

Steward decision modules remain separate:

- `collisionSteward.js`
- `trackLimitsSteward.js`
- `pitLaneSpeedingSteward.js`
- `tireRequirementSteward.js`
- `penaltyLedger.js`

### `src/simulation/vehicle/contactResolution.js`

Owns:

- generic collision-pair resolution loop
- overlap correction
- pit-controlled collision displacement rule
- contact velocity response
- collision cooldown updates

It may import `collisionGeometry.js`, but `raceSimulation.js` should not own the physical response math.

### `src/simulation/vehicle/runoffResponse.js`

Owns:

- runoff detection response
- off-track speed damping
- heading correction toward the track
- wheel-surface refresh after correction

Track-limit stewarding stays in `src/simulation/rules/trackLimitsSteward.js`; runoff response is vehicle behavior, not penalty behavior.

### `src/simulation/snapshots/raceSnapshots.js`

Owns:

- full public snapshot assembly
- render snapshot assembly
- observation snapshot assembly
- snapshot dependency object construction
- winner snapshot formatting

The low-level car and pit serializers remain in:

- `src/simulation/vehicle/vehicleSnapshots.js`
- `src/simulation/pit/pitSnapshots.js`

### `src/simulation/race/raceOrder.js`

Owns:

- ordered-car sorting
- driver race context
- aggression calculation
- DRS reference car lookup if it remains coupled to order/gap state

Timing math stays in `src/simulation/timing/`.

### `src/simulation/race/raceControlState.js`

Owns:

- initial race-control object creation
- initial safety-car state creation
- start-light timing metadata
- race-control reset helper if restart code needs it later

Lifecycle updates stay in `raceLifecycle.js`; safety-car movement stays in `safetyCar.js`; classification stays in `classification.js`.

### `src/simulation/race/raceDistance.js`

Owns small pure race-distance utilities:

- total-lap normalization
- progress wrapping
- forward distance
- signed progress delta
- lap calculation
- finish-distance calculation

These are utilities, not orchestration. They should be easy to test without constructing a full race.

### `src/simulation/rules/penaltyStats.js`

Owns:

- empty penalty-stat shape
- penalty-stat aggregation by driver
- penalty-stat lookup with default

`raceSimulation.js` should ask for penalty stats but not build maps itself.

### `src/simulation/race/raceProgress.js`

Owns:

- per-car progress refresh from wheel/track state
- lap telemetry reset/update calls
- sector performance refresh
- timing sample and timing-line crossing refresh
- rank/gap/attack-state field assignment
- DRS latch update coordination

It should not own driver AI, physics integration, or public snapshot assembly.

### `src/simulation/race/raceFinish.js`

Owns:

- detecting newly finished cars
- emitting `car-finish`
- invoking tire-requirement review for finishers
- applying outstanding service penalties after all cars finish
- building final classification through `classification.js`
- switching race control into finished safety-car mode
- emitting `race-finish`

Classification ranking math remains in `classification.js`.

### `src/simulation/race/redFlag.js`

Owns:

- red-flag hold application for cars during `step()`
- clearing attack/DRS state while red flag is active

Red-flag state toggling remains in `safetyCar.js` or the existing race-control lifecycle module unless a later cleanup separates it further.

## Refactor Contract

- Preserve `createRaceSimulation()`.
- Preserve `F1RaceSimulation`.
- Preserve `snapshot()`.
- Preserve `snapshotRender()`.
- Preserve `snapshotObservation()`.
- Preserve `consumeStepEvents()`.
- Preserve package exports and types.
- Preserve environment `reset()` and `step()` return shapes.
- Preserve race behavior under current tests and browser smoke.
- Preserve current docs that mark weather, reliability, and fuel-load behavior as reserved future modules.
- Every extracted function should receive `sim` plus narrow domain arguments, or a clearly named state object. Do not introduce hidden global mutable state.
- Facade methods may remain in `F1RaceSimulation` when they are public or used by tests, but their bodies should delegate to feature-owned modules.
- Small pure helpers should move to utility modules instead of staying as anonymous top-level functions in `raceSimulation.js`.
- After this plan, top-level helper functions in `raceSimulation.js` should be limited to `createRaceSimulation()` and truly package-local glue that cannot sensibly belong anywhere else.
- Domain constants should move with the module that owns the behavior. `raceSimulation.js` may keep package-level constants such as `FIXED_STEP`, but pit, collision, race-distance, and penalty constants should not remain there after their owning modules are extracted.
